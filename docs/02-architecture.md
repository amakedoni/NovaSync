## Ключевые архитектурные решения

### 1. Почему Electron, а не Tauri

| Критерий | Electron | Tauri |
|----------|----------|-------|
| Прозрачные окна | `transparent: true` — из коробки | Сложно, особенно на Windows |
| Глобальные хоткеи | `globalShortcut.register` — из коробки | Нужен плагин или Rust-код |
| Multi-window | Нативно, IPC между окнами | Сложнее |
| Скриншоты | `desktopCapturer` — из коробки | Нужен отдельный крейт |
| Размер сборки | ~150 MB | ~10 MB |
| Потребление RAM | ~150 MB в покое | ~50 MB |

**Для десктопного ассистента, где прозрачные окна и хоткеи — ключевой функционал, Electron выигрывает.** Tauri можно рассмотреть на V2.

### 2. Одно окно или два?

**Для MVP — ОДНО окно**, которое анимированно меняет позицию и размер:

| Режим | Позиция | Размер |
|--------|---------|--------|
| `idle/input` | Центр экрана | 600×400 |
| `response` | Правый нижний угол | 400×500 |

**Для V0.2+ — ДВА окна:**
- **Overlay** (центр): появляется по хоткею, исчезает после отправки запроса
- **Response Widget** (правый нижний угол): живёт пока пользователь не закроет

Причина разделения на V0.2: оверлей не должен висеть пока идёт ответ, а виджет должен оставаться доступным даже когда пользователь вызывает оверлей для нового вопроса.

### 3. Два React-приложения (V0.2+)

- `src/overlay/` → Overlay React tree
- `src/widget/` → Widget React tree

Причины разделения:
- Изоляция стилей (разный CSS, размеры, поведение)
- Отдельный lifecycle (оверлей живёт секунды, виджет — минуты)
- Vite multi-page через `rollupOptions.input`

### 4. LLM-запросы из main process, а не renderer

```
┌──────────┐    IPC     ┌──────────┐    IPC     ┌──────────┐
│ Overlay  │◄──────────►│  Main    │◄──────────►│ Response │
│ Renderer │            │ Process  │            │ Widget   │
└──────────┘            └──────────┘            └──────────┘
                              │
                              ▼
                        ┌──────────┐
                        │ LLM API  │
                        │(external)│
                        └──────────┘
```

- API-ключи не светятся в devtools renderer'а
- Не блокирует UI
- Можно добавить кеширование/retry logic централизованно

## Взаимодействие с системой

### Модель безопасности (три уровня)

```
┌─────────────────────────────────────────────────────┐
│  УРОВЕНЬ 1: READ-ONLY (всегда включено)              │
│  • clipboard.read (текст)                            │
│  • screen.capture (ручной, через хоткей)              │
│  • window.getActive (title + process name)           │
│  • env.read (определённые переменные)                 │
│  • fs.listDir (только чтение списка файлов)           │
├─────────────────────────────────────────────────────┤
│  УРОВЕНЬ 2: CONFIRMABLE (подтверждение пользователя)  │
│  • fs.writeFile (только в ~/NovaSync/workspace)      │
│  • browser.openUrl                                   │
│  • clipboard.write                                   │
│  • process.launch (белый список программ)             │
├─────────────────────────────────────────────────────┤
│  УРОВЕНЬ 3: DANGEROUS (выключены по умолчанию)        │
│  • shell.exec (произвольные команды)                   │
│  • fs.delete                                         │
│  • network.fetch (произвольные хосты)                  │
│  • system.settings                                   │
└─────────────────────────────────────────────────────┘
```

### Техническая реализация (Electron)

**Буфер обмена:**
```typescript
// electron/main/services/clipboard.ts
import { clipboard } from 'electron';

export const readClipboard = (): string => clipboard.readText();
export const writeClipboard = (text: string): void => clipboard.writeText(text);
```

**Скриншот:**
```typescript
// electron/main/services/screenshot.ts
import { desktopCapturer, screen } from 'electron';

export const captureScreen = async (): Promise<Buffer> => {
  const sources = await desktopCapturer.getSources({
    types: ['screen'],
    thumbnailSize: screen.getPrimaryDisplay().workAreaSize,
  });
  return sources[0].thumbnail.toPNG();
};
```

**Активное окно (Windows):**
```powershell
powershell -command "Get-Process | Where-Object {$_.MainWindowTitle -ne ''} | Select-Object MainWindowTitle, ProcessName"
```

## Как дать ассистенту «зрение»

### Подход A: Ручной скриншот (MVP V0.2)
- Пользователь нажимает кнопку «📸» в оверлее
- Electron делает скриншот через `desktopCapturer`
- PNG base64 отправляется в multimodal-модель (GPT-4o-mini / Gemini Flash)
- **+** Дёшево, надёжно, пользователь контролирует
- **–** Не continuous, требует действия

### Подход B: OCR + контекст окна (локально, без LLM)
- Electron получает скриншот
- Tesseract.js (бесплатный OCR) извлекает текст
- Текст + заголовок + имя процесса → в системный промпт
- **+** Полностью локально, приватно
- **–** Картинки/диаграммы теряются

### Подход C: Проактивный анализ (V2, дорого)
- Раз в N секунд скриншот → дешёвая модель → «что на экране?»
- Результат как контекст
- **+** Максимальный контекст
- **–** Дорого, нагрузка на систему

### Рекомендация для MVP
**Подход A + элементы B:**
- Кнопка «Захватить экран» в оверлее
- Всегда: имя активного окна + заголовок (бесплатно, нет API-затрат)
- OCR — только по явному запросу
